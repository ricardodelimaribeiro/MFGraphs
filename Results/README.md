# Benchmark Results

This folder contains generated performance analysis reports and benchmark outputs. Files are auto-generated by test and profiling scripts â€” do not edit manually.

## Benchmark Output Files

### benchmark_latest.{csv,json}
- **Generated by:** `Scripts/BenchmarkSuite.wls`
- **Purpose:** Raw benchmark data from test runs across all solvers and cases
- **Refresh frequency:** After running `BenchmarkSuite.wls` (auto-generated)
- **Format:** Timestamped copies also created (e.g., `benchmark_2026-04-01_153045.csv`)
- **Contents:** Test case, tier, solver type, timing, memory, status, residual
- **Usage:** Load into analysis tools or spreadsheet for performance trending

### compare_dnf.md
- **Generated by:** `Scripts/CompareDNF.wls` (with or without `--tag`)
- **Purpose:** Compare custom `DNFReduce` vs Mathematica's built-in `BooleanConvert` algorithm
- **When to check:**
  - After optimizations to `DNFReduce.wl` (to verify improvement)
  - When evaluating new reduction strategies
  - As part of performance regression detection
- **Key metrics:**
  - **Speedup:** `BooleanConvert` time / `DNFReduce` time (> 1 means DNFReduce is faster)
  - **Disjunct counts:** Number of Or-branches in output (fewer is better for subsequent Reduce calls)
  - **Leaf counts:** Expression complexity (fewer leaves = simpler expressions)
  - **Equivalence:** Whether both methods produce logically equivalent results

### jamarat_phase_points_latest.{csv,json}
- **Generated by:** `Scripts/JamaratFeasibilityPhaseDiagram.wls`
- **Purpose:** Pointwise feasibility labels for Jamaratv9 phase diagrams used in appendix-style plots
- **Default sweeps:** `I1/I2` split at fixed total, cost-difference grid (`U2-U1`, `U3-U1`), and one-parameter windows for `U1`, `U2`, `U3`
- **Key columns:** `Slice`, `SweepVariable`, `I1`, `I2`, `U1`, `U2`, `U3`, `Delta21`, `Delta31`, `Feasible`, `Status`, `ResultKind`

### jamarat_phase_boundaries_latest.{csv,json}
- **Generated by:** `Scripts/JamaratFeasibilityPhaseDiagram.wls`
- **Purpose:** Derived feasible intervals per sweep (for text tables and captions)
- **Key columns:** `Slice`, `Variable`, `FeasibleIntervalsText`, sampled/feasible/infeasible counts

### jamarat_phase_costdiff_matrix_latest.csv
- **Generated by:** `Scripts/JamaratFeasibilityPhaseDiagram.wls`
- **Purpose:** Heatmap-ready matrix (`Delta31` rows, `Delta21` columns, values in `{0,1}` for infeasible/feasible)

### jamarat_phase_metadata_latest.json
- **Generated by:** `Scripts/JamaratFeasibilityPhaseDiagram.wls`
- **Purpose:** Run configuration, point counts, and per-slice feasibility summary for reproducibility

### bottleneck_report.md
- **Generated by:** `Scripts/BottleneckReport.wls`
- **Purpose:** Detailed timing breakdown identifying where solver time is spent
- **When to run:** After making changes to `DNFReduce.wl`, `DataToEquations.wl`, or `NonLinearSolver.wl`
- **What it shows:**
  - Call count and cumulative time for each profiled function
  - Breakdown of solver stages (preprocessing, DNFReduce, TripleClean, etc.)
  - Which bottleneck is most significant for each test case
- **How to interpret:**
  - High `DNFReduce` time â†’ focus on Boolean algebra optimizations
  - High `TripleClean` time â†’ fixed-point iteration is taking many steps
  - High `FindRoot` time â†’ numerical solving is expensive; consider `PrecomputeM` interpolation
- **Related documentation:** See `BENCHMARKS.md` Â§ "Identified bottlenecks" for context

### reduction_strategies_profile.md
- **Generated by:** Custom profiling within `BenchmarkSuite.wls` (optional)
- **Purpose:** Benchmark different disjunct reduction strategies on cases with high Or-branch counts
- **What it compares:**
  1. **Subsumption Pruning** â€” pairwise implication checks (removes subsumed branches)
  2. **Full Reduce** â€” uses Mathematica's `Reduce` on all branches
  3. **Group & Merge** â€” groups similar branches and combines them
- **When to review:**
  - When disjunct count is large (> 200, per `$ReduceDisjunctsThreshold`)
  - When evaluating new reduction methods
  - To understand tradeoffs between speed and result simplicity
- **Key columns:** Case, strategy, time, disjunct count, leaf count, quality metric

## Performance History Files

These files are auto-appended by benchmark scripts and track optimization impact over time.

### DNF_PERFORMANCE_HISTORY.md
- **Purpose:** Historical record of every optimization to `DNFReduce`
- **Append frequency:** After running `Scripts/CompareDNF.wls --tag "description"`
- **How it works:**
  1. Run `CompareDNF.wls` with a tag describing your change
  2. Script automatically measures speedup across 8 test cases
  3. Results formatted and appended to this file with git commit hash
- **Usage:** Review to understand what optimizations help which cases
- **Location:** Root directory (`DNF_PERFORMANCE_HISTORY.md`)

### PARALLEL_PERFORMANCE_HISTORY.md
- **Purpose:** Historical record of parallelization impact
- **Append frequency:** After running `Scripts/BenchmarkSuite.wls [tier] --tag "description"`
- **Metrics:** Wall-clock time per solver, per tier, with kernel count noted
- **Usage:** Understand when parallel dispatch provides benefit vs. overhead
- **Location:** Root directory (`PARALLEL_PERFORMANCE_HISTORY.md`)

## Workflow: Running and Interpreting Benchmarks

### 1. Establish baseline before changes
```bash
wolframscript -file Scripts/BenchmarkSuite.wls small --tag "baseline"
# Runs small tier, appends entry to PARALLEL_PERFORMANCE_HISTORY.md
```

### 2. Make your changes

### 2a. (Optional) Generate Jamarat phase-diagram data
```bash
wolframscript -file Scripts/JamaratFeasibilityPhaseDiagram.wls profile=quick
# For appendix-quality sweeps:
wolframscript -file Scripts/JamaratFeasibilityPhaseDiagram.wls profile=paper
```

### 3. Re-benchmark to measure impact
```bash
wolframscript -file Scripts/BenchmarkSuite.wls small --tag "my optimization"
# Appends new entry with your git commit hash
```

### 4. Compare results
- Open `Results/benchmark_latest.csv` and filter for your two runs
- Or check `PARALLEL_PERFORMANCE_HISTORY.md` for the appended entries
- Speedup = baseline time / new time (> 1 means improvement)

### 5. Run profiling to understand where time went
```bash
wolframscript -file Scripts/BottleneckReport.wls
# Generates Results/bottleneck_report.md with function-level breakdown
```

### 6. Commit benchmark entry
The history files are part of the repository â€” commit them alongside your code changes to document the performance impact.

## Troubleshooting Benchmark Runs

### Benchmark timeout or hangs
- Check if solver is still running: `Activity Monitor` or `ps aux | grep wolframscript`
- If stuck, review `Results/bottleneck_report.md` to identify slow cases
- Try running a single case with custom timeout:
  ```bash
  wolframscript -file Scripts/BenchmarkSuite.wls core case=7 timeout=600
  ```

### Results look wrong or unexpected
- Verify git status is clean (no uncommitted changes)
- Check if `$MFGraphsVerbose = True` is affecting timing (should be `False` for benchmarks)
- Verify recent code changes didn't break solvers; run `Scripts/RunTests.wls fast` first

### Can't find expected output file
- Files are created in `Results/` (not current directory)
- Timestamped files use format: `benchmark_YYYY-MM-DD_HHMMSS.csv`
- Latest run always accessible via `benchmark_latest.csv`
