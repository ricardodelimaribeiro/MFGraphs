# Block-ordering benchmark — findings (PR 3 / Phase B)

Generated by `Scripts/Research/block_ordering/benchmark_orderings.wls`.

## What was measured

For each of 37 registered scenarios (Grid1020 skipped — kernel crashes on
that one even outside this experiment), `activeSetReduceSystem` was timed
under each of four `"DisjunctOrdering"` choices:

- `Lexicographic` — the existing default (sort Or conjuncts by structural form).
- `Block-Vertex`  — group disjuncts by anchor vertex; BFS order from boundary.
- `Block-Edge`    — group by anchor edge; BFS from boundary edges.
- `Block-SCC`     — group by SCC; topological order, vertex order within SCC.

Anchor convention: junction vertex `v` for `Or[j[a,v,b]==0, …]`; head vertex
`b` for `Or[j[a,b]==0, …]`. Edge counterparts: `{v,b}` and `{a,b}`.

Per-call timeout 60 s; raw rows in `block_ordering/benchmark.csv`.

## Hard-case behavior

Scenarios that hit the 60 s timeout under Lexicographic continue to time
out under all Block-* orderings: case_21, case_23, HRF_Scenario_1,
Grid0505, Grid0707, Grid0710, Grid1010. **No new scenario is cracked**
by block ordering alone, but Grid0404 — the one previously-cracked grid
that still took 30 s — drops by 5 s (~17%).

## Takeaway (Decision section)

Block ordering wins broadly:
- 21 of 37 scenarios show measurable speedup (>5%) under at least one Block-*.
- 0 of 37 scenarios show a regression >10%.
- The biggest absolute wins (Achdou_2023_junction 4.95×, Grid0404 −5 s)
  come from the same family of orderings: **Block-Edge and Block-SCC**
  outperform Block-Vertex on cracked-but-slow cases, while Block-Vertex
  is competitive on tiny cases.

**Recommendation**: ship `"DisjunctOrdering"` as a public opt-in (already
done in this PR; default stays Lexicographic). A follow-up PR can promote
Block-Edge or Block-SCC to default after a per-scenario regression review
on the full benchmark suite (BENCHMARKS.md). Picking between Edge and SCC
is a coin flip on this data.

## Sensitivity sweep (Phase A.3) — orthogonal finding

The sensitivity sweep ran on 22 scenarios (those with ≤30 disjuncts AND
baseline solve <10 s; raw rows in `sensitivity/*.json`). For each Or, we
solved the system with the disjunct removed (replaced by `True`) and with
each branch forced individually.

Across **323 disjuncts × 646 forced branches: every modified solve produced
the same flow-variable values as the baseline** (L∞ delta = 0). On these
scenarios, the disjuncts don't shape the *answer* — they shape the *search*.
The deterministic constraints (Kirchhoff + boundary + accumulated equality
rules) already pin the solution; the complementarities are redundant
relative to the final answer but the solver still has to traverse them.

This is consistent with the branching results from PR 2: the most-influential
disjuncts have high TopShare not because they encode independent answer
information, but because the search expands more sub-branches before
collapsing them. Block ordering helps because it surfaces the redundancy
sooner — neighboring disjuncts share variables and substitute each other
out within a vertex / edge / SCC block.

**Limitation**: this finding is on the *cracked-and-small* subset. The hard
cases (Grid* >= 5×5, HRF, case_21, case_23) have >30 disjuncts and were
excluded. Whether their disjuncts are similarly answer-redundant is open;
running the sweep on them would require solving each modified system
to completion, which the baseline can't do.
